## 购物广场 PRD

## 1. 文档定位

- 模块名称：购物广场
- 所属项目：多 Agent 家居电商 Demo
- 模块级定位：总项目中的 **P1 展示型范围**
- 首期目标：做一个可演示、可录屏、可承接主链路的“内容化商品发现页”，不阻塞核心对话式下单链路

## 2. 背景与价值

根据根目录现有文档，购物广场不是传统电商“纯商品列表页”，而是：

- 商品发现入口
- Agent 战绩展示区
- 个性化推荐橱窗
- 从“逛”平滑过渡到“让 AI 帮我配”的导流页

它的价值不是取代核心对话页，而是补足“逛”的体验，让用户在 Demo 中看到：

- 平台有持续更新的商品与内容
- Agent 在真实地帮用户发现 deal
- 页面具备千人千面潜力
- 用户可从广场快速进入主链路

## 3. 目标与非目标

### 3.1 模块目标

1. 让用户快速理解“购物广场 = Agent 驱动的精选发现页”
2. 展示今日上新、促销、风格推荐、Agent 省钱战绩
3. 支持基于会话偏好的轻量个性化排序
4. 为核心链路导流：点击商品/专题/战绩卡后可进入 `/chat`
5. 满足录屏演示与静态展示需要

### 3.2 非目标

- 不做全网实时商品抓取
- 不做真实社区发帖/评论系统
- 不做真实商家入驻与广告系统
- 不做完整购物车闭环
- 不做复杂推荐算法平台化能力

## 4. 用户与场景

### 4.1 目标用户

- 对家居搭配感兴趣、但不想自己逐个筛选商品的用户
- 已进入过主链路、具备基础偏好标签的回访用户
- 投资人/合作方/客户，用于快速理解产品想象空间

### 4.2 核心场景

1. 用户进入广场，看见“今日 Agent 战绩”和精选商品
2. 用户浏览自己关心的品类，如沙发、床垫、餐桌
3. 用户点击某张卡片，进入带预置意图的对话页
4. 用户在广场看到结构化评价与促销信息，建立信任感

## 5. 核心体验定义

购物广场首期建议为单页面 `/plaza`，由以下区域组成：

### 5.1 顶部 Banner

- 传达“AI 正在为你发现更优家居 deal”
- 展示当日主题，如“今日客厅焕新”“猫家庭耐抓推荐”
- 提供主 CTA：`让 AI 帮我搭配`

### 5.2 今日上新 / 今日促销

- 数据来源：`homary.json`
- 展示商品卡：图片、标题、类目、价格、标签
- 可打上 `New`、`Hot`、`Deal`、`Agent 推荐` 标签
- 首期以规则排序和静态分组为主

### 5.3 个性化推荐区

- 基于 session 中的兴趣标签轻量排序
- 示例：偏好沙发的用户优先看到沙发与客厅组合
- 如果没有偏好数据，则退化为默认精选内容

### 5.4 Agent 战绩流

- 用内容卡展示“Agent 帮别人省了多少钱/完成了什么搭配”
- 示例：`Alice 的 Agent 刚为她搭配了一套现代极简书房，立省 $250`
- 首期可基于模板 + 商品数据生成，不要求真实用户产生

### 5.5 结构化评价展示

- 展示商品的结构化亮点，而非长文本评论
- 示例字段：材质、风格、物流时效、适用空间、耐抓/环保标签
- 可加入“Agent 总结”短句，提升可信度

### 5.6 智能唤醒卡片

- 展示“可被 Agent 主动提醒的后续需求”
- 示例：`您昨天看过猫抓布沙发，需要我继续帮您找搭配茶几吗？`
- 首期以静态模板或 session 标签触发为主

### 5.7 导流机制

- 商品卡 CTA：`让 AI 讲讲这件商品`
- 战绩卡 CTA：`生成类似方案`
- 类目卡 CTA：`按这个风格继续挑`
- 点击后跳转到 `/chat`，并带上预填 query / 标签

## 6. 功能需求

### 6.1 P0（广场模块首期必做）

1. `/plaza` 页面基础布局
2. Banner + 主 CTA
3. 今日上新 / 今日促销商品区
4. Agent 战绩流静态展示
5. 结构化评价卡片展示
6. 轻量个性化排序（基于 session 标签）
7. 广场到 `/chat` 的导流跳转
8. 空状态、加载态、错误态

### 6.2 P1（模块增强项）

1. 定时上新机制
2. 更细粒度的个性化推荐
3. 智能唤醒卡片动态化
4. “查看更多同风格方案”专题页
5. 与长期记忆增强联动

## 7. 数据与技术约束

### 7.1 数据来源

- 商品主数据：当前目录 `homary.json`
- 偏好标签：当前 session / Redis
- 战绩流与唤醒卡：模板生成或规则生成

### 7.2 技术栈对齐

- 前端：Next.js 14 App Router
- 后端：FastAPI
- 状态/缓存：Redis
- 数据与检索：PostgreSQL / pgvector
- 实时扩展：WebSocket（非首期强依赖）
- Agent 编排：LangGraph（仅在与主链路联动时使用）

## 8. 页面信息架构

1. 顶部导航 / 返回主站入口
2. Banner
3. 个性化推荐区
4. 今日上新区
5. 今日促销区
6. Agent 战绩流
7. 结构化评价区
8. 智能唤醒区
9. 底部 CTA：进入对话式搭配

## 9. 验收标准

### 9.1 产品验收

- 页面能体现“Agent 驱动的购物广场”定位
- 能看见上新、促销、战绩、评价四类内容
- 点击至少 3 类卡片都能导流到 `/chat`
- 无偏好用户和有偏好用户都能看到合理内容

### 9.2 技术验收

- 页面首屏可正常加载
- 商品数据可从 `homary.json` 或后端接口稳定读取
- Session 偏好标签能影响展示顺序
- 页面在 Demo 模式下可稳定录屏展示

## 10. 风险与降级

### 10.1 风险

- 个性化能力过重，拖慢首月 Demo 节奏
- 广场功能过多，偏离主链路
- 缺少真实用户内容，导致社区感不足

### 10.2 降级策略

- 首期以静态区块 + 规则排序为主
- 战绩流使用模板生成内容
- 唤醒卡片先做展示，不接真实消息系统
- 广场只负责“展示 + 导流”，不承担完整交易闭环
