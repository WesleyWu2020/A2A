# PRD：多 Agent 家居电商 Demo

## 1. 文档信息

- 产品名称：多 Agent 驱动的 AI 家居电商 Demo
- 文档版本：V1.0
- 文档类型：首月 Demo 产品需求文档（PRD）
- 面向对象：产品、设计、前端、后端、AI、测试、业务展示团队
- 产品形态：Web 演示站点

## 2. 项目背景

传统家具/家居电商在重决策、多 SKU 采购场景中效率较低。用户通常需要在多个商品页、平台和标签页之间反复比价、核对尺寸、确认材质、查找优惠，完成一套房间或全屋采购往往需要花费大量时间。

本项目希望通过 LLM + Multi-Agent System，将“商品搜索”升级为“需求理解 + 方案生成 + 自动议价 + 履约跟踪”的新一代 AI 购物体验，并以首月 Demo 的形式对美国市场展示核心理念。

## 3. 产品愿景

打造一个由买家 Agent 和卖家 Agent 协同驱动的 AI 电商平台，让用户通过一句自然语言需求，获得可视化的空间搭配方案、议价结果和一键下单体验。

## 4. 产品目标

### 4.1 Demo 目标

1. 跑通“输入需求 → Agent 工作 → 输出 3 套方案 → 一键下单”的核心链路。
2. 展示买家 Agent 与卖家 Agent 的协同与议价过程。
3. 让用户直观感知“更省时间、更省钱、更懂需求”的产品价值。
4. 形成可录屏、可演示、可对外讲解的高保真 Demo。

### 4.2 用户价值目标

#### 买家侧

- 将复杂家居采购时间压缩到分钟级体验。
- 从商品推荐升级为“整套空间解决方案”。
- 让用户可视化看到 Agent 的工作过程、议价过程与决策理由。

#### 卖家侧

- 展示“零代码接入 Agent”的概念能力。
- 展示“设置底价和库存策略后，由卖家 Agent 自动参与撮合”的机制。

## 5. 目标用户

### 5.1 核心用户

1. 美国市场的家具/家居消费者
   - 需要布置客厅、卧室、书房等空间
   - 有预算、风格、尺寸、材质、耐用性等多维要求
   - 对整套方案和时间效率敏感

2. Demo 观看者
   - 潜在合作方、投资人、商家、媒体
   - 更关注产品差异化、技术可行性和商业想象空间

### 5.2 次级用户

- 卖家角色（本期可为模拟角色）

## 6. 核心使用场景

用户输入自然语言需求，例如：

> “预算 3000 美元，帮我搭配一套原木风客厅家具，家里有两只猫，沙发需要耐抓。”

系统完成以下步骤：

1. 理解需求并补充提问
2. 生成结构化用户画像
3. 检索当前目录 `homary.json` 中的 4500 个 Homary 商品候选数据
4. 调用买家 Agent 组装方案，并调用卖家 Agent 模拟议价
5. 输出 3 套带布局建议的组合方案
6. 展示 Agent 活动日志、议价记录和决策理由
7. 用户确认其中 1 套方案并一键下单

## 7. MVP 范围

### 7.1 P0 必做范围

#### 买家端核心体验

- 极简首页与 `Try Now` 入口

- 2 分钟预估进度反馈
- 3 套方案结果页

- 建立演示级议价规则

#### Agent 演示能力

- 买家 Agent：需求理解、商品筛选、组合推荐、方案解释

### 7.2 P1 展示型范围

- 购物广场
- 卖家零代码控制台
- 长期记忆增强体验
- 结构化评价体系展示

### 7.3 非本期范围

- 全网实时商品搜索
- 真实商家入驻闭环

## 8. 关键功能需求

## 8.1 首页

### 目标

清晰表达产品价值，并引导用户进入 Demo 核心链路。

### 需求

- 展示“AI 帮你完成复杂家居采购”的核心卖点
- 保留单一主 CTA：`Try Now`
- 首页首期使用 2D 视觉进行展示，如平面空间示意图、商品卡片拼贴、2D 布局效果图
- 本期首页不引入 3D 场景、3D 交互或实时 3D 渲染能力
- 页面适合对外录屏与展示

## 8.2 对话式需求采集

### 目标

以自然对话替代表单，降低输入成本。

### 需求

- 支持用户一句话输入需求
- 支持追问预算、房间、风格、面积、特殊限制
- 将需求沉淀为结构化 Profile
- 支持中途修改，如“沙发改成深色”

## 8.3 Agent 活动日志

### 目标

让用户看到 Agent 的工作过程，增强信任感和效率感。

### 需求

- 日志展示至少包含检索、过滤、询价、议价、汇总等阶段
- 每个阶段有可读文案
- 支持与主流程同步更新

## 8.4 方案结果页

### 目标

向用户交付可比较、可理解、可决策的完整方案。

### 需求

- 固定输出 3 套方案
- 每套方案包含：
  - 方案名称
  - 风格说明
  - 商品列表
  - 原价 / 折后价
  - 推荐理由
  - 空间效果图或布局建议

## 8.5 议价记录

### 目标

展示 Agent 的差异化能力与省钱价值。

### 需求

- 展示起始价格、优惠策略、最终价格
- 展示卖家 Agent 的让利原因，如库存压力、新客优惠
- 议价结果需与方案价格一致

## 8.6 一键下单与履约

### 目标

完成从推荐到交易的闭环演示。

### 需求

- 支持选择某套方案后下单
- 生成演示订单号
- 展示订单成功页
- 展示履约节点卡片，如已下单、备货中、已发货、预计送达

## 11. 成功指标

### 产品体验指标

- 用户可在 1 次点击内进入核心体验
- 方案生成体验总时长控制在 2 分钟级别
- 用户可明确看到 Agent 工作过程

### 演示指标

- 主链路稳定跑通率 ≥ 95%
- 适合录制对外演示视频
- 页面文案、视觉、流程具备较强故事性

### 业务表达指标

- 清晰表达“比传统电商提效 10 倍”的概念
- 清晰表达“多 Agent 议价与撮合”的差异化价值

## 12. 非功能需求

- 稳定性：核心链路必须稳定可演示
- 性能：页面响应快，加载和等待时有明确反馈
- 一致性：页面风格、Agent 语气、状态文案统一
- 可扩展性：后续可接入更多商品源、卖家和真实交易能力

## 13. 风险与应对

### 风险

1. 多 Agent 流程过于复杂，首月难稳定
2. 商品数据标签不完善，影响推荐质量
3. 实时生成内容不稳定，影响 Demo 连贯性
4. 可视化效果图产出耗时

### 应对策略

1. 聚焦核心链路，控制范围
2. 对关键结果采用规则 + 模板兜底
3. 购物广场和卖家控制台以展示页为主
4. 空间可视化使用静态高质量效果图
