多Agent 驱动的AI电商平台项目计划书



一、 核心目标与价值主张

- 产品定位：处理复杂购物场景的基于LLM和多智能体协同（Multi-Agent System）的双边市场的电商平台。

- 买家侧感知：“这是一个由 AI 大模型和买家、卖家双边智能体驱动的新一代购物网站。它懂我需求，能快速展示 2D 空间搭配效果与布局建议，还会帮我寻找更优 deal，并跟踪完成交易履约，是一个多 SKU 采购管家。”

  - 核心交付体验：1，美国消费者在全屋装修、多房间采购这类重决策家居消费中，从搜索商品到比价下单通常要花 20-30 小时，我们希望大幅提升复杂采购效率，给出可视化解决方案，节约消费者 80% 时间，减少搜索、比价、找折扣券、追踪履约和售后的时间。2，可以定时推荐商品或商品组合；2-1，在页面中生成商品组合在客户空间中的 2D 效果图或布局建议，帮助用户快速决策；3，买家 Agent 可以和卖家 Agent 进行沟通；4，买家 Agent 跟踪完成耗时耗力的订单查询、售后客服等；5，有一个活的社区，用户感受到 Agent 是活的、智能的，而且越来越智能；

- 卖家侧：目前痛点是广告费贵，佣金高，运营操作上下架、广告投放、客服耗时，人力成本高。

  - 从致欧等上市家具跨境公司财报来看，广告费占销售收入15-20%，导致公司净利润低（致欧4%净利），促销效果不稳定。

  - 对卖家的价值是：1，低成本新渠道，AI匹配买家无广告费，节约80%费用，和90%的人工成本。3，AI的数据中台可以给选品策略、爆品策略、利润检测、SKU 组合销售/搭配优化、销售预测、退货率预测等；4，用户决策路径长（需要大量产品信息 & 社会证明）。我们可以提供高质量内容、社区、真实的使用评测和新一代评价体系

  - 未来开放API给卖家

  - 在卖家的产品上面直接搭建一层 Agent，这样买家的 Agent 可以直接和卖家的 Agent 进行沟通，减少卖家的广告投放；



二、 首月 Demo 交付物--对美国市场产品展示

基于这个理念，我们想要做一个 Demo 产品对美国市场进行展示。主要包括的内容有：

- 我们已经准备了 Homary 的 4500 个商品数据，当前以目录下的 `homary.json` 作为首月 Demo 的商品数据源；

- 设计一个高保真可交互原型 / 跑通核心链路的 Web 网站：包含前端、后端及买卖双边 Agent 的协作体验。用户可以输入“预算 3000 刀，帮我配齐一套原木风客厅家具，要适合有两只猫的家庭，沙发需要耐抓”等需求，系统在当前目录 `homary.json` 的 4500 个 Homary 商品中完成检索、组合、议价与结果展示。初期基于封闭沙盒确保演示稳定性，暂不接入全网实时搜索。

- 交互流程为：用户在页面输入需求（比如“预算 3000 刀，帮我配齐一套原木风客厅家具，要适合有两只猫的家庭，沙发需要耐抓”），买家 Agent 开始与卖家 Agent 进行沟通，完成商品查询、优惠讨论和方案整理。用户在等待数分钟后，可以看到 Agent 输出的 3 套带布局建议的组合方案，并进行确认和一键下单。用户还可以查看 AI Agent 的搜索过程、议价记录和决策过程。

- 概念演示视频 (Demo Video)：

 页面开发完成后，我们使用录屏工具录制一个概念演示视频。

     对比组 (2小时)：在Amazon这个平台和homary.com这个大型家具家居独立站搜索关键词 -> 在几十页列表中迷失 -> 开多个标签页比对尺寸/材质 -> 计算满减 -> 耗时20分钟以上

     实验组 (10分钟)：输入一句话需求 -> 等待 Agent 工作完成 -> 回来查看 Agent 整理的 3 套带 2D 布局建议/空间效果图的组合方案 -> 确认最优解并查看 Agent 砍价记录 -> 一键下单。耗时是对比组的 10%

- 首月 Demo 技术实现遵循统一技术栈：前端采用 Next.js 14 App Router，后端采用 FastAPI，多 Agent 编排采用 LangGraph，商品与检索数据存储在 PostgreSQL / pgvector，会话与缓存使用 Redis，Agent 活动日志通过 WebSocket 实时推送。

     

三、 模块重构与功能细化 (MVP 范围)

模块 1：买家端交互层

- 首页体验：极简首页，强化“对话即服务”。点击 "Try Now" 进入分屏模式（左侧：2D 视觉化的商品/方案流；右侧：对话与 Agent 状态流）。首页首期不引入 3D 场景、3D 交互或实时 3D 渲染。

- 无感 Onboarding (破冰)：

     优化建议：不要一次性抛出所有表单（名字、预算、面积等），而是通过对话引导。例如：“我是你的专属家居买手，请问今天想改造哪个房间？” 在多轮对话中自然补全用户画像（Profile）。AI等主动问名字，年龄、性别、爱好、名字面积、装修风格，预算范围，需要的deadline 等

- “活的” Agent 状态流 (核心亮点)：

    设计理念：为了让用户感受到 10 倍提效和省钱，必须把 Agent 的“工作过程”可视化。

    UI 呈现：在对话框上方或侧边栏增加“Agent 活动日志”（Timeline），例如：

    “正在全库扫描原木风双人沙发 (已过滤 3,200 件不符商品)...”

    “发现目标商品，正在与卖家 Agent 询价...”

    “谈判成功！基于您的新用户身份，成功向卖家申请到额外 5% 折扣。”

- 一键履约：方案生成后，提供“一键生成订单”按钮，后续的物流节点通过 Agent 以卡片形式主动推送。



模块 2：购物广场

- 形态转变：这不仅是商品流，更是“Agent 的战绩展示区”。

- 这个广场里面每天定时上新商品，并且是千人千面的，例如我关心沙发，那么我看到的主要是沙发；我关心床垫，那么我看得主要是今天上新的床垫或者开始做促销的床垫

- 内容呈现：

    展示如：“Alice 的 Agent 刚刚为她搭配了一套现代极简书房，立省 $250”。

- 人机共创评价体系：Agent 自动抓取商品参数和履约时长生成客观评价（如：物流提前2天到达，实物色差<5%），用户只需点击“Approve（认可）”并附上一张实拍图。这种评价比传统纯人工评价更具结构化和可信度。

- 智能唤醒：每天下午 2 点，Agent 不是生硬地推销，而是基于长尾需求发问：“您昨天确定的沙发下周二到，需要我帮您看看匹配的抱枕吗？”



模块 3：会话记忆与长期记忆增强

- P0 会话记忆 (Session Context)：处理当前对话的上下文，随时接受用户的修改（“那个沙发换成深色的”）。

- P1 长期记忆增强 (User Graph)：建立用户的专属数据库（Vector DB）。记录审美偏好（现代、复古）、空间限制（客厅只有 20 平米）、特殊标签（有猫、对甲醛敏感）、历史客单价。

- 演示亮点（P1）：第二次打开网站时，Agent 可以直接说：“欢迎回来，您之前选过猫抓布沙发，这次要看卧室家具吗？”



模块 4：核心撮合引擎 (产品与多 Agent 博弈)

- 首月降级策略：为了首月能拿出 Demo，暂不实时检索全网，将当前目录 `homary.json` 中的 4500 个 Homary 商品作为平台的“第一方数据库”，并写入 PostgreSQL / pgvector 支持检索与过滤。

- 双 Agent 博弈机制 (重点技术攻关)：

    买家 Agent (Demand)：目标是“性价比最高、最符合记忆标签”。

    卖家 Agent (Supply)：目标是“促成交易、清理库存、保证最低利润率”。

- 业务逻辑：买家 Agent 选定商品后，在后台通过 API 与卖家 Agent 握手。卖家 Agent 根据设定的底价和库存压力，给出动态折扣（如：库存多时给予 15% 优惠券）。



模块 5：卖家零代码控制台

- 一键入驻：卖家只需上传商品图片、基础参数和**“价格底线 (Bottom Price)”**。

- AI 自动完善：平台 AI 自动生成多语种（一开始英语和中文就行）的高转化商品描述、SEO 标签、并提取材质和尺寸特征存入向量库。

- 自动营销：卖家不需要去设置复杂的满减活动。只要告诉自己的 Agent：“这批库存我想在两周内清掉，最低可以接受 8 折”。当有买家 Agent 来询价时，卖家 Agent 会自动进行讨价还价。



1. 在这个 Demo 中，只是为了演示这个流程，不需要真实的买家和卖家。一切目的是为了服务我们展示自己的设计理念。

2. 进度预期管理的部分，可以设计一个 2 分钟的预估时间进度条；空间可视化建议在 Demo 中使用精心制作的静态效果图、平面空间示意图或 2D 布局建议，而非实时渲染，质量更可控。

3. 首月 Demo 不接真实支付与真实物流系统，交易闭环以“一键下单 + 订单成功页 + 履约状态模拟”完成展示。

Demo 范围收敛的建议不变——聚焦核心链路，购物广场和卖家控制台可以做成静态页面点缀，不需要真正跑通逻辑。
